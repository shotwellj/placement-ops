## Next session: Phase B3 plan

Cluster JDs to discover emerging role archetypes (Talent Engineer, Prompt Engineer, Embedded ML Engineer, Data Engineer→Full-Stack hybrid, etc.).

### Key design questions to resolve before building

1. **Clustering algorithm.** Options: k-means on skill vectors, hierarchical clustering on co-occurrence, LLM-as-judge clustering. My lean: hybrid — embed JDs with skill vectors, k-means for grouping, LLM to NAME the resulting clusters.
2. **What we actually surface.** A "discovered archetype" needs a name, a skill profile, example reqs. Where does it live in the schema?
3. **How does it feed back into matching.** Does an archetype affect adjacency weights? Does it change how candidates get scored?

### Pre-flight before B3

We need MORE THAN 7 reqs to cluster meaningfully. Either:
- Run intake on 5-10 more diverse JDs first
- Defer B3 until natural usage produces more reqs
- Accept that early clustering will be noisy and iterate

### Recommendation

Don't build B3 next. Build the **resolver fix** (post-processing fuzzy match for raw_skill_text) instead. It will boost resolution rate to ~60-70% on existing data without needing manual taxonomy work, and it's a 1-2 hour build. THEN do B4 (skill mesh) which uses that cleaner data. THEN B3.

Better next-session order: resolver fix → B4 → B3.

## Closing update (end of session)

After writing the journal above, two more changes shipped:

### Resolver upgrade (api/_compliance.py)

Added two tiers to `resolve_skill_id`:
- **Tier 3:** Check `skill_resolution_decisions` for previous user decisions. The B2 approval queue now permanently trains the resolver — every alias decision becomes a future automatic resolution.
- **Tier 4:** Single-token whole-word fallback for raw texts of 1-3 tokens. Conservative; bails out on ambiguous matches and on long phrases (4+ tokens) to avoid false positives.

One-shot backfill via `scripts/backfill_skill_resolution.py` ran the new resolver against existing rows. Spot-checked all resolutions for false positives — clean.

### GET `/api/reqs` endpoint (api/index.py)

Pipeline list view was silently broken (404'd) since the page was built. Endpoint now exists with optional `?status=` filter and a per-req submission_count subselect to avoid N+1 calls. Also removed 16 lines of unreachable dead code from inside `list_submissions` that was the orphaned remains of an earlier refactor.
