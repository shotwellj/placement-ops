## Next session: Phase B3 plan

Cluster JDs to discover emerging role archetypes (Talent Engineer, Prompt Engineer, Embedded ML Engineer, Data Engineer→Full-Stack hybrid, etc.).

### Key design questions to resolve before building

1. **Clustering algorithm.** Options: k-means on skill vectors, hierarchical clustering on co-occurrence, LLM-as-judge clustering. My lean: hybrid — embed JDs with skill vectors, k-means for grouping, LLM to NAME the resulting clusters.
2. **What we actually surface.** A "discovered archetype" needs a name, a skill profile, example reqs. Where does it live in the schema?
3. **How does it feed back into matching.** Does an archetype affect adjacency weights? Does it change how candidates get scored?

### Pre-flight before B3

We need MORE THAN 7 reqs to cluster meaningfully. Either:
- Run intake on 5-10 more diverse JDs first
- Defer B3 until natural usage produces more reqs
- Accept that early clustering will be noisy and iterate

### Recommendation

Don't build B3 next. Build the **resolver fix** (post-processing fuzzy match for raw_skill_text) instead. It will boost resolution rate to ~60-70% on existing data without needing manual taxonomy work, and it's a 1-2 hour build. THEN do B4 (skill mesh) which uses that cleaner data. THEN B3.

Better next-session order: resolver fix → B4 → B3.

## Grandmaster tier — 5 features shipped (session continues into evening)

After the billing reset, built five new intake features in sequence to bring
the JD parser output from "junior sourcer in a box" to "grandmaster in a box"
before launching on LinkedIn.

### What shipped

1. **Alternative titles (Feature 1)** — expanded alt_titles schema from a
   flat level list to three dimensions: level_progression (same role at
   different IC levels), functional_aliases (same person, different naming
   convention at peer companies), adjacent_crossover (different roles where
   the same person could shift, with easy/moderate/hard transition difficulty).
   UI: title-intelligence card with three sub-sections. Backwards-compat with
   old flat-shape reqs.

2. **Skill alternatives (Feature 2)** — separate LLM call that takes each
   must-have skill and returns 2-4 functional equivalents (Spark ->
   Snowpark, Trino, Databricks DLT) with transferability rating
   (high=day-1 ready, medium=phone screen, low=ramp-up). Non-fatal on
   failure. UI card shows each skill with color-coded pills.

3. **Watering holes (Feature 3)** — embedded in the JD parser prompt
   itself rather than a separate call. Five archetype-specific examples
   in the prompt (embedded, ML/AI, security, finance eng, defense/cleared).
   Eight venue_types: mailing_list, conference, community, publication,
   code_host, training_alumni, competition, discord_slack. Target 5-8
   venues spanning 3+ types. UI card with icons per venue type and a
   how-to-use code block per venue.

4. **Objection playbook (Feature 4)** — separate LLM call that generates
   3-5 predictable candidate objections for THIS specific role+comp+
   company+location combination with pre-written 2-4 sentence counters.
   Seven objection categories: industry_perception, comp_below_market,
   location_remote, brand_unknown, career_risk, visa_clearance_blocker,
   tech_stack_skepticism. Counter has Copy button (reuses existing
   .bool-copy handler). Confidence level (high/medium/low) per objection.

5. **Sequenced 21-day play (Feature 5)** — separate LLM call that takes
   parsed context PLUS the already-extracted Tier 1 companies and
   watering holes and produces a 5-phase sequenced outreach play.
   Phases: warm-channel opener (days 1-3, 30-50% response), Tier 1
   cold + personalization (days 4-7, 8-15%), Tier 2 broader outreach
   (days 8-14, 3-7%), unconventional + watering holes (days 15-21,
   10-20%), back-channels (days 22+). UI renders as vertical timeline
   with color-coded phase badges (warm->cool: green, blue, amber,
   purple, slate).

### Intake flow now

Five LLM calls serially:
1. JD parser (now includes watering_holes in the same call)
2. Boolean builder
3. Skill alternatives (non-fatal)
4. Objection playbook (non-fatal)
5. Sequenced play (non-fatal)

Expected wall time: 30-60s. All three new calls are non-fatal — if they
time out or fail, the intake still ships the parser + booleans and
simply omits the card that didn't generate. If this becomes a UX
problem we'll parallelize with asyncio.gather, but serial failure modes
are cleaner for now.

### Storage

All five new fields live in the existing requisitions.parsed_json blob.
No schema migration. UI reads from parsed.{alt_titles, skill_alternatives,
watering_holes, objection_playbook, sequenced_play}. Intake response also
returns them as top-level fields for immediate render without re-fetch.

### UI order in renderDetail

Comp -> Skills -> **AltTitles** -> **SkillAlternatives** -> Boolean ->
**WateringHoles** -> Submissions -> Evaluate -> **ObjectionPlaybook** ->
**SequencedPlay** -> Notes -> RawJD

### Pro tier decision

Originally planned to gate features 1-5 behind a $49/mo Pro tier.
Reversed that decision during the build: features 1-5 are now FREE
and included in every intake for every user. Reasoning:

  - The entire LinkedIn launch strategy depends on recruiters saying
    "this is genuinely useful" on first use. Putting the best features
    behind a paywall kills that
  - The 5-intake/month free cap still protects against runaway costs
  - Pro tier will be "unlimited intakes + deeper features we build
    next month" (layoff intel, company deep-dive, cross-company
    calibration data) — things that DON'T exist yet
  - Free tier being genuinely great is the differentiator vs Gem/hireEZ
